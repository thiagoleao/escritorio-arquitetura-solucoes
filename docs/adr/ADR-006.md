# ADR-006 - Geração Padronizada de Artefatos por Atividade

**Status:** Proposta (revisado — unifica a proposta original com o ADR-007)
**Data:** 15 de julho de 2026

## Contexto

Atividades de um planejamento de arquitetura frequentemente envolvem produzir um artefato concreto — um diagrama, uma ADR, um documento de integração (INT) — ou coletar insumos de descoberta, mapeamento e reuniões. Hoje nada disso é suportado pela aplicação: esses artefatos são produzidos inteiramente fora dela.

Uma restrição importante orienta este ADR: **não delegar a geração completa do documento/diagrama ao modelo de linguagem livremente**. Pedir a um LLM para "desenhar um diagrama C4" ou "escrever uma ADR" do zero, a cada execução, produz resultados com estrutura e nível de detalhe diferentes toda vez — inconsistente entre atividades, entre planejamentos e entre empresas. A aplicação deve, em vez disso, ter **templates estruturais fixos** por tipo de artefato, e usar o modelo de linguagem apenas para **preencher conteúdo dentro dessa estrutura** — nunca para decidir layout, seções ou estilo.

**Nota de revisão:** a versão original deste ADR previa classificar cada atividade manualmente (campo `activity_type` preenchido só pelo usuário), registrando como débito técnico que isso deveria futuramente ser automático. Uma proposta separada (ADR-007) chegou a ser rascunhada só para resolver esse débito. Ao revisar as duas juntas, ficou claro que implementar a classificação manual primeiro e trocá-la pela sugestão automática logo em seguida seria retrabalho — a versão abaixo já nasce com o mecanismo automático, unificando as duas propostas num único ADR. O ADR-007 foi descontinuado e seu conteúdo incorporado aqui.

## Pré-requisito antes de iniciar a implementação

Esta ADR **não deve começar a ser implementada** até que os templates estruturais fixos estejam definidos e solicitados previamente:
- XML compatível com draw.io para o diagrama de fluxo e para os 4 níveis do C4 Model (Contexto, Container, Componente, Código);
- Template de documento (seções, ordem, formatação) para ADR e para o documento de integração (INT).

Esses templates são a peça central da decisão de não gerar diagrama/documento livremente via LLM (ver Contexto) — sem eles definidos antes, a implementação não tem em que preencher o conteúdo gerado pelo modelo.

## Decisão

### 0. Entregáveis catalogados como insumo para classificação da atividade

Para saber se uma atividade "envolve diagrama", "envolve ADR" etc., a aplicação precisa de um sinal — e o momento mais natural para capturar esse sinal é quando o próprio usuário já descreve, na tela de geração, quais entregáveis a solução precisa.

**Catálogo de entregáveis conhecidos (fixo no código, não uma tela de administração nesta fase):**

| Chave | Rótulo | `activity_type` correspondente |
|---|---|---|
| `diagrama_arquitetura` | Diagrama de arquitetura (fluxo ou C4) | "envolve diagrama" |
| `documento_adr` | Documento ADR | "envolve ADR" |
| `documento_int` | Documento de integração (INT) | "envolve documento de integração" |

Na tela de criação do planejamento (`src/app/page.tsx`), enquanto o usuário preenche os dados necessários para gerar o planejamento (empresa, contexto, objetivo, entregáveis, restrições), ele também marca — ao lado do campo livre "Entregáveis esperados", que continua existindo e sendo enviado ao LLM como hoje — checkboxes para os itens do catálogo que essa solução precisa. Essa seleção fica em uma nova coluna `plannings.deliverable_types JSONB NOT NULL DEFAULT '[]'` — atributo do planejamento, não versionado por `planning_versions` (mesmo tratamento já dado a `deliverables`/`context`/`objective`/`constraints`).

O prompt do LLM passa a receber a lista de `deliverable_types` selecionados, com instrução explícita: ao gerar cada atividade, se o propósito dela for produzir um dos entregáveis catalogados marcados, preencher `activity_type` já com o valor correspondente. **Isso é sugestão automática, não vínculo definitivo** — o campo continua editável manualmente pelo usuário (mesmo padrão de campo editável já usado para classificação de feedback), para corrigir os casos em que o modelo erra o vínculo atividade↔entregável, ou para atividades adicionadas manualmente depois da geração inicial (que não passam pelo prompt, então não têm sugestão automática).

Atividades de descoberta/mapeamento/reunião (item 4 abaixo) não têm um entregável catalogado equivalente — continuam dependendo de classificação manual, já que não há um "produto" com template para servir de sinal.

### 1. Geração de diagramas

Atividades com `activity_type` = "envolve diagrama" ganham um botão "Gerar diagrama" no card. Modelos suportados inicialmente:
- Diagrama de fluxo;
- C4 Model, nos 4 níveis: Contexto, Container, Componente e Código.

Cada tipo de diagrama tem um template estrutural fixo (posicionamento, estilos, formas pré-definidas) em XML compatível com draw.io. O modelo de linguagem recebe esse template e só preenche o conteúdo variável (nós, rótulos, relações, textos) — nunca gera o XML do zero nem decide layout. Saída: arquivo `.drawio`, aberto diretamente no draw.io sem necessidade de ajuste manual de formatação.

### 2. Geração de ADR / documento de integração (INT)

Atividades com `activity_type` = "envolve ADR" ou "envolve documento de integração" ganham um botão de geração equivalente. Mesmo princípio do item 1: um template de documento fixo (seções, ordem, formatação) onde o modelo preenche o conteúdo de cada seção. Formato de saída (`.docx` ou markdown) e ferramenta de geração a definir na fase de design técnico desta ADR.

### 3. Identidade visual por empresa

Os dois geradores (diagrama e documento) permitem trocar **apenas o logotipo** (e possivelmente uma cor de destaque) por empresa, sem alterar a estrutura do template. Implica associar um asset de logotipo a cada `company` já existente no modelo de dados.

### 4. Upload de insumos por atividade

Atividades classificadas como "descoberta", "mapeamento" ou "reunião" ganham um botão de upload de arquivo no card, associando o(s) arquivo(s) àquela atividade específica — diferente do upload já existente no ADR-001, que é sobre o planejamento como um todo, não sobre uma atividade individual. Essa classificação continua manual (item 0 acima), já que não há entregável catalogado equivalente.

## Riscos e considerações

- **Esforço de design técnico dos templates C4 é o maior risco de escopo desta ADR.** Desenhar layouts compatíveis com draw.io que fiquem visualmente corretos nos 4 níveis do C4 Model é trabalho de design técnico não trivial, provavelmente maior que a integração do LLM em si.
- **Acerto do LLM ao vincular atividade → entregável catalogado não é garantido.** O modelo pode gerar uma atividade cujo propósito real é produzir o diagrama, mas não marcar `activity_type` corretamente, ou marcar uma atividade errada. Classificação manual continua disponível como correção — o mecanismo reduz o atrito médio, não o elimina.
- **Catálogo fixo no código.** Adicionar um novo tipo de entregável exige alteração de código (nova entrada no catálogo + template correspondente), não uma tela de admin — mesmo raciocínio de simplicidade já usado em outras enumerações do sistema (`VALID_STATUSES`, `VALID_CLASSIFICATIONS`): baixo volume de tipos, mudança pouco frequente.
- **Reaproveitamento de ferramentas existentes.** A geração de documentos Word pode reaproveitar o skill de geração de `.docx` já disponível no ambiente de desenvolvimento, reduzindo o esforço da geração de ADR/INT.
- **Onde o novo relacionamento `activity -> attachment` vive.** Diferente do upload do ADR-001 (por requisição, descartado), este upload precisa persistir associado à atividade — implica nova tabela no `planner-api`, análoga a `activity_feedback` mas para arquivos.
- **Coluna `deliverable_types` não versionada.** Editar os entregáveis catalogados depois da primeira geração não dispara nova sugestão automática de `activity_type` para atividades já existentes — só influencia gerações futuras (nova versão), a menos que o usuário reclassifique manualmente.

## Fora de escopo

Edição visual dos diagramas dentro da própria aplicação (a edição continua no draw.io, fora da aplicação); templates de diagrama além de fluxo e C4 (ex.: diagrama de sequência, modelo entidade-relacionamento) — podem ser adicionados depois seguindo o mesmo padrão de template fixo; OCR de arquivos enviados (já fora de escopo desde o ADR-001); geração automática de todo o conteúdo de um artefato sem revisão humana — os artefatos gerados continuam sendo rascunhos a revisar, mesmo com estrutura padronizada; catálogo de entregáveis editável por tela de administração; re-sugestão automática de `activity_type` ao editar um planejamento existente (a sugestão só acontece na geração inicial via LLM).

## Critérios de sucesso

- Ao criar um planejamento, o usuário consegue marcar quais entregáveis catalogados (diagrama, ADR, INT) a solução precisa, além de continuar descrevendo outros livremente em texto.
- Planejamentos gerados com entregáveis catalogados marcados retornam as atividades correspondentes já com `activity_type` sugerido, sem exigir classificação manual do zero na maioria dos casos.
- Uma atividade com `activity_type` = "envolve diagrama" mostra o botão de geração correspondente e produz um `.drawio` que abre sem erros e visualmente consistente no draw.io.
- Uma atividade com `activity_type` = "envolve ADR" ou "envolve documento de integração" mostra o botão de geração correspondente e produz um documento com estrutura fixa e conteúdo preenchido pelo modelo.
- Trocar o logotipo cadastrado de uma empresa muda a identidade visual de todos os artefatos gerados para ela, sem exigir alteração no template.
- Uma atividade de descoberta/mapeamento/reunião (classificada manualmente) permite anexar um ou mais arquivos, e o conteúdo extraído fica associado a ela (consultável depois).
- O usuário consegue corrigir manualmente a classificação de qualquer atividade quando o modelo erra ou quando não há entregável catalogado envolvido.

# ADR-004 - Board Interno de Acompanhamento de Execução

**Status:** Aprovado
**Data:** 15 de julho de 2026

## Contexto

O ADR-001 previa, como evolução futura, integração com o Trello para acompanhamento das atividades depois de geradas. O ADR-002 fechou o ciclo de planejamento (gerar → revisar → aprovar), mas nenhuma das duas ADRs cobre o que acontece **depois** de um planejamento aprovado: a execução real das atividades hoje acontece fora da aplicação, sem nenhuma visão consolidada de progresso.

A decisão tomada é **não** integrar com uma ferramenta de terceiros. Em vez disso, a aplicação passa a ter um board Kanban interno, simples, que agrega um ou mais planejamentos aprovados e mostra em que estágio de execução cada atividade está.

Este ADR é o primeiro de uma família maior de melhorias já identificadas (login/sessão por usuário, hospedagem pública do frontend, redesign visual, drag-and-drop, geração padronizada de diagramas e documentos ADR/INT por atividade, upload de insumos por atividade). Essas ficam registradas como backlog para ADRs futuras — este documento cobre somente o board de acompanhamento.

## Decisão

### 1. Escopo do board

O board mostra planejamentos com `status = 'approved'` — este é o critério de "projeto ativo". Planejamentos ainda em geração, revisão ou revisados (mas não aprovados) não aparecem; planejamentos arquivados somem do board.

O usuário pode:
- ver todos os projetos ativos ao mesmo tempo, agrupados por empresa;
- filtrar para ver apenas os projetos ativos de uma empresa específica.

### 2. Colunas e nomenclatura

Três colunas, mesmo padrão de nomenclatura já usado para o status do planejamento (valor interno em inglês, rótulo em português):

- `todo` → **A Fazer**
- `doing` → **Em Andamento**
- `done` → **Concluída**

A movimentação entre colunas nesta primeira versão acontece por um controle simples (botão/seletor), não por arrastar-e-soltar — drag-and-drop fica para uma ADR futura.

### 3. Status de execução é desacoplado do versionamento

Esta é a decisão técnica central do ADR. A tabela `activities` é **versionada**: cada nova versão de um planejamento (ADR-002, Fase 2) recria as linhas de atividades do zero, mesmo quando o conteúdo não muda. Se o status de execução fosse uma coluna em `activities`, marcar uma atividade como concluída e depois revisar o planejamento (criando a versão seguinte) apagaria esse progresso silenciosamente, porque a nova versão recria as linhas sem o valor.

Por isso, o status de execução é armazenado numa tabela própria, `activity_execution`, chaveada por `(planning_id, activity_external_id)` — independente de qual seja a versão atual do planejamento. O board sempre lê o conteúdo (título, marco) da versão atual e junta com o status de execução dessa tabela separada. Revisar um planejamento nunca afeta o progresso de execução já registrado.

### 4. Histórico de mudanças de status

Toda mudança de status é registrada em `activity_status_history` (de/para, com timestamp). Não há uso dessa tabela na interface desta primeira versão, mas o custo de gravar é baixo e deixa a base pronta para métricas futuras de tempo de execução (já mencionadas como evolução prevista no ADR-002).

### 5. Percentual de conclusão

Cada planejamento exibido no board mostra o percentual de atividades concluídas em relação ao total de atividades da versão atual.

## Modelo de dados

```sql
activity_execution(
  id, planning_id, activity_external_id,
  status ('todo'|'doing'|'done'), updated_at,
  unique(planning_id, activity_external_id)
)

activity_status_history(
  id, planning_id, activity_external_id,
  from_status, to_status, changed_at
)
```

## Consequências positivas

- Fecha o ciclo planejamento → execução dentro da própria aplicação, sem depender de ferramenta externa.
- Visão consolidada de progresso entre múltiplos projetos e empresas.
- Progresso de execução sobrevive a revisões do planejamento (decisão de desacoplamento).
- Base de dados já criada (`activity_status_history`) para métricas de tempo de execução futuras.

## Consequências negativas

- Mais uma dimensão de estado para manter consistente (execução, separada do conteúdo do planejamento).
- Sem drag-and-drop nesta versão, a movimentação entre colunas é menos fluida que um Kanban tradicional — aceito como simplificação deliberada.

## Fora de escopo (backlog para ADRs futuras)

Login e sessão por usuário; hospedagem pública do frontend (hoje acessado via túnel `gcloud run services proxy`); drag-and-drop; redesign visual completo; geração padronizada de diagramas (C4/drawio) e documentos ADR/INT por atividade; upload de insumos por atividade.

## Critérios de sucesso

- Aprovar um planejamento faz com que ele apareça no board.
- Arquivar um planejamento o remove do board.
- Mover uma atividade entre colunas atualiza o percentual de conclusão do seu planejamento.
- Filtrar por empresa mostra somente os planejamentos aprovados daquela empresa; sem filtro, mostra todos, agrupados por empresa.
- Revisar um planejamento (criar nova versão) não apaga o status de execução das atividades já em andamento ou concluídas.

# ADR-001 - Arquitetura do MVP do Planejador de Soluções de Arquitetura

**Status:** Proposta
**Data:** 14 de julho de 2026

## Contexto

O trabalho de arquitetura de soluções normalmente começa com informações dispersas, como contexto do problema, objetivo da entrega, documentos, restrições, requisitos, padrões técnicos e dados fornecidos por diferentes pessoas.

Antes de produzir diagramas, documentos ou decisões técnicas, é necessário organizar essas informações e definir:

- os marcos necessários para concluir a arquitetura;
- a sequência recomendada de trabalho;
- as atividades que precisam ser executadas;
- as dependências entre as atividades;
- as informações ainda ausentes;
- os pontos que bloqueiam a evolução da entrega.

Atualmente, esse planejamento é feito manualmente pelo arquiteto. Isso exige esforço recorrente para interpretar os insumos, estruturar o trabalho e criar as atividades em uma ferramenta de gestão, como o Trello.

A solução proposta deve inicialmente resolver apenas essa etapa.

O usuário fornecerá:

- contexto da demanda;
- objetivo da solução;
- entregáveis esperados;
- informações técnicas;
- restrições;
- dados textuais;
- arquivos de apoio.

A aplicação deverá analisar esses insumos e produzir:

- um roadmap para conclusão da solução de arquitetura;
- uma lista organizada de atividades;
- os marcos da entrega;
- as dependências entre as atividades;
- os bloqueios identificados;
- as perguntas que precisam ser respondidas;
- uma sequência sugerida de execução.

Neste primeiro momento, o usuário transferirá manualmente as atividades para o Trello. A integração automática com o Trello será considerada em uma evolução posterior.

## Problema

É necessário definir uma arquitetura inicial que permita validar rapidamente a utilidade da solução, sem introduzir banco de dados, gestão de usuários, integrações externas ou mecanismos complexos de workflow.

O MVP precisa ser simples de implementar, testar e modificar.

## Decisão

Será desenvolvida uma aplicação web stateless, sem banco de dados, composta por uma interface única e uma API responsável por processar os insumos e gerar o roadmap.

A solução terá inicialmente os seguintes componentes:

### 1. Interface web

A interface permitirá ao usuário:

- descrever o contexto da demanda;
- informar o objetivo;
- informar os entregáveis esperados;
- informar restrições e observações;
- anexar arquivos;
- solicitar a geração do planejamento;
- visualizar o roadmap;
- visualizar a lista de atividades;
- copiar o conteúdo gerado.

A primeira versão poderá ser construída como uma página única.

### 2. API de aplicação

A API será responsável por:

- receber os campos textuais;
- receber os arquivos anexados;
- validar os formatos e tamanhos permitidos;
- extrair o conteúdo textual dos arquivos;
- organizar os insumos em um formato padronizado;
- montar a instrução enviada ao modelo de linguagem;
- solicitar a análise ao modelo;
- validar a estrutura da resposta;
- devolver o resultado para a interface.

### 3. Serviço de processamento de arquivos

O processamento dos arquivos ocorrerá apenas durante a requisição. Os arquivos serão mantidos temporariamente apenas pelo tempo necessário para extração e análise. Após a conclusão ou falha da requisição, os arquivos temporários deverão ser removidos.

A primeira versão deverá priorizar formatos textuais de fácil extração, como:

- PDF com texto;
- TXT;
- Markdown;
- DOCX.

Arquivos digitalizados que dependam de OCR não fazem parte do escopo inicial.

### 4. Integração com modelo de linguagem

A geração do roadmap será realizada por meio de uma chamada a um modelo de linguagem.

A aplicação enviará ao modelo:

- contexto consolidado;
- objetivo;
- entregáveis;
- restrições;
- conteúdo extraído dos arquivos;
- instruções de geração;
- esquema esperado para a resposta.

O modelo deverá retornar uma resposta estruturada.

A aplicação não deverá depender diretamente de um único fornecedor. A integração com o modelo será encapsulada em uma interface interna, permitindo substituir o provedor posteriormente.

Exemplo conceitual:

```
LLMProvider
  generateArchitecturePlan(input)
```

Implementações futuras poderão utilizar diferentes provedores sem alterar a lógica principal da aplicação.

### 5. Resposta estruturada

A resposta do modelo deverá seguir um contrato previamente definido. O formato inicial deverá conter:

```json
{
  "summary": "Resumo do entendimento da demanda",
  "assumptions": [
    "Premissas identificadas"
  ],
  "missing_information": [
    "Informações ainda necessárias"
  ],
  "milestones": [
    {
      "id": "M1",
      "title": "Nome do marco",
      "objective": "Objetivo do marco",
      "completion_criteria": [
        "Critério de conclusão"
      ]
    }
  ],
  "activities": [
    {
      "id": "A1",
      "title": "Nome da atividade",
      "description": "Descrição da atividade",
      "milestone_id": "M1",
      "dependencies": [],
      "status": "ready",
      "expected_output": "Resultado esperado"
    }
  ],
  "blockers": [
    {
      "description": "Descrição do bloqueio",
      "related_activity_ids": [
        "A2"
      ]
    }
  ]
}
```

A resposta poderá ser apresentada de forma visual na interface, mas o contrato entre a API e a interface será estruturado em JSON.

### 6. Ausência de persistência

O MVP não terá banco de dados. A aplicação não armazenará:

- projetos;
- usuários;
- históricos;
- arquivos;
- roadmaps anteriores;
- respostas do modelo;
- atividades concluídas.

Ao atualizar ou fechar a página, o conteúdo poderá ser perdido. O usuário será responsável por copiar o resultado e transferir manualmente as atividades para o Trello ou outra ferramenta.

### 7. Execução síncrona

A geração será inicialmente realizada de forma síncrona. O usuário enviará os dados, aguardará o processamento e receberá o resultado na mesma interação.

Não serão utilizados inicialmente:

- filas;
- workers;
- processamento assíncrono;
- notificações;
- callbacks;
- webhooks.

Caso o processamento ultrapasse limites aceitáveis ou passe a envolver arquivos maiores, essa decisão deverá ser revisitada.

## Fluxo principal

```
Usuário
  |
  | Preenche contexto, objetivo e entregáveis
  | Anexa arquivos
  v
Interface web
  |
  | Envia formulário
  v
API
  |
  | Valida os dados
  | Extrai o conteúdo dos arquivos
  | Consolida o contexto
  | Monta a instrução
  v
Modelo de linguagem
  |
  | Gera roadmap estruturado
  v
API
  |
  | Valida e normaliza a resposta
  v
Interface web
  |
  | Exibe marcos, atividades, bloqueios e pendências
  v
Usuário
  |
  | Copia manualmente as atividades para o Trello
```

## Escopo do MVP

Faz parte do MVP:

- formulário para entrada textual;
- upload de arquivos;
- extração de texto dos formatos suportados;
- chamada ao modelo de linguagem;
- geração de roadmap;
- geração de marcos;
- geração de atividades;
- identificação de dependências;
- identificação de bloqueios;
- identificação de informações ausentes;
- visualização do resultado;
- opção para copiar o conteúdo;
- tratamento básico de erros.

## Fora do escopo do MVP

Não faz parte da primeira versão:

- autenticação;
- múltiplos usuários;
- banco de dados;
- histórico de projetos;
- edição colaborativa;
- integração com Trello;
- criação automática de cards;
- acompanhamento da execução;
- mudança de status das atividades;
- notificações;
- geração automática de diagramas;
- geração completa de documentos arquiteturais;
- agentes autônomos;
- memória entre execuções;
- OCR;
- processamento assíncrono;
- busca vetorial;
- base de conhecimento;
- integração com repositórios corporativos.

## Consequências positivas

- reduz o tempo necessário para colocar a primeira versão em funcionamento;
- permite validar o valor da solução antes de construir integrações;
- reduz o custo de infraestrutura;
- simplifica implantação e manutenção;
- evita decisões prematuras sobre persistência;
- facilita alterações no formato do roadmap;
- permite testar diferentes modelos e instruções;
- mantém o foco no problema principal;
- cria uma separação clara entre interface, aplicação e provedor de IA.

## Consequências negativas

- o usuário poderá perder o resultado ao fechar a página;
- não haverá histórico;
- não será possível retomar automaticamente um projeto;
- não haverá acompanhamento das atividades;
- o usuário precisará cadastrar manualmente as tarefas no Trello;
- arquivos precisarão ser reenviados a cada nova análise;
- requisições com arquivos grandes poderão levar mais tempo;
- o processamento síncrono poderá limitar a escalabilidade inicial;
- a qualidade da saída dependerá da qualidade dos insumos e das instruções enviadas ao modelo.

## Riscos

### Resposta inconsistente do modelo

O modelo pode devolver uma resposta diferente do contrato esperado.

Mitigação:
- solicitar resposta estruturada;
- utilizar schema validation;
- rejeitar ou corrigir respostas inválidas;
- registrar erros técnicos sem armazenar o conteúdo dos documentos;
- criar testes com diferentes tipos de demanda.

### Roadmap genérico

O modelo pode gerar atividades superficiais ou excessivamente genéricas.

Mitigação:
- separar claramente contexto, objetivo, entregáveis e restrições;
- definir critérios mínimos para cada atividade;
- exigir dependências e resultados esperados;
- utilizar exemplos de referência;
- evoluir as instruções com base no uso real.

### Excesso de confiança na saída

O roadmap pode conter premissas incorretas apresentadas como fatos.

Mitigação:
- separar fatos, premissas e informações ausentes;
- obrigar o modelo a declarar incertezas;
- exibir aviso de que o resultado precisa ser revisado pelo arquiteto;
- não permitir que informações inferidas sejam tratadas como decisões aprovadas.

### Exposição de informações sensíveis

Os arquivos podem conter dados internos ou confidenciais.

Mitigação:
- não persistir os arquivos;
- remover arquivos temporários;
- utilizar conexão criptografada;
- restringir logs;
- não registrar o conteúdo completo das solicitações;
- utilizar um provedor de IA compatível com as políticas de cada contexto de uso.

## Alternativas consideradas

### Aplicação com banco de dados desde o início

Não escolhida porque adicionaria:

- modelagem de projetos;
- autenticação;
- controle de acesso;
- persistência de arquivos;
- migrações;
- backup;
- governança de dados.

Esses elementos não são necessários para validar a principal hipótese do produto.

### Integração com Trello no MVP

Não escolhida porque exigiria:

- autenticação OAuth;
- gestão de credenciais;
- escolha de workspace, quadro e lista;
- tratamento de duplicidade;
- sincronização de status;
- tratamento de falhas externas.

A integração será desenvolvida somente depois que o formato das atividades estiver validado.

### Uso de n8n como aplicação principal

Não escolhido como primeira opção para a interface e a lógica central.

O n8n poderá ser utilizado posteriormente para integrações e automações, mas a lógica principal do produto deverá permanecer em uma aplicação controlada pelo projeto. Isso evita que o contrato do produto, o processamento dos arquivos e a validação da resposta fiquem excessivamente acoplados a uma ferramenta de automação.

### Arquitetura baseada em agentes

Não escolhida.

O MVP possui um fluxo previsível:

```
Receber dados
Extrair conteúdo
Montar contexto
Chamar modelo
Validar resposta
Exibir resultado
```

Não há necessidade inicial de autonomia, planejamento iterativo ou coordenação entre agentes.

### Processamento assíncrono

Não escolhido por aumentar a complexidade operacional.

Será reavaliado caso:

- os arquivos se tornem maiores;
- o tempo de processamento seja elevado;
- existam múltiplas etapas de análise;
- seja necessário acompanhar o status da execução.

## Critérios de sucesso

A decisão será considerada adequada se o MVP conseguir:

- receber uma demanda real de arquitetura;
- processar texto e arquivos;
- gerar marcos coerentes com o objetivo;
- gerar atividades acionáveis;
- apresentar dependências entre as atividades;
- destacar informações ausentes;
- identificar atividades bloqueadas;
- produzir conteúdo que possa ser transferido para o Trello;
- permitir ajustes rápidos na instrução e no contrato de saída;
- funcionar sem persistência de dados.

## Critério para revisão desta ADR

Esta ADR deverá ser revisada quando ocorrer uma das seguintes situações:

- necessidade de salvar projetos;
- necessidade de consultar histórico;
- integração com Trello;
- múltiplos usuários;
- autenticação;
- processamento de arquivos grandes;
- necessidade de filas e workers;
- acompanhamento do progresso;
- geração iterativa do roadmap;
- uso de conhecimento específico por empresa;
- exigência de auditoria;
- necessidade de armazenar decisões e artefatos.

## Evolução prevista

A evolução seguinte deverá considerar:

- persistência de projetos;
- autenticação;
- integração com Trello;
- criação automática de cards;
- atualização dos cards quando o roadmap mudar;
- acompanhamento do progresso;
- registro de decisões arquiteturais;
- base de conhecimento por empresa ou projeto.

## Decisão resumida

O primeiro produto será uma aplicação web stateless, sem banco de dados e sem integração com o Trello. Ela receberá contexto, objetivo, entregáveis, informações textuais e arquivos. Uma API extrairá e organizará o conteúdo, enviará os dados para um modelo de linguagem e devolverá um roadmap estruturado com marcos, atividades, dependências, bloqueios e informações pendentes. O resultado será revisado pelo arquiteto e transferido manualmente para o Trello.

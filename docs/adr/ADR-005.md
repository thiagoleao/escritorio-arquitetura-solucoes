# ADR-005 - Autenticação, Sessão e Experiência do Board

**Status:** Aprovado — implementado (itens 1–5 concluídos em 15/07/2026)
**Data:** 15 de julho de 2026

## Contexto

Hoje a aplicação roda restrita por IAM do Google Cloud Run (`--no-allow-unauthenticated`) — só a conta Google do usuário tem acesso, e o acesso local exige manter um túnel (`gcloud run services proxy`) ativo na máquina. Não existe conceito de usuário dentro da própria aplicação: `planning_versions.created_by` guarda só `'model'` ou `'user'`, genérico.

Isso é adequado para uso solo, mas tem três limitações que o uso real já expôs: (1) depender de um túnel manual para acessar a aplicação é frágil e ficou sujeito a expirar sessões de autenticação do `gcloud` no meio do uso; (2) não há como saber *quem* fez uma edição ou aprovação quando mais de uma pessoa usar a aplicação; (3) a experiência de front-end ainda é mínima — sem persistência de estado ao atualizar a página, sem drag-and-drop no board (ADR-004) e sem um design system definido.

Este ADR agrupa as melhorias de acesso e experiência que ficaram registradas como backlog nas ADRs anteriores.

## Decisão

### 1. Login e sessão por usuário

A aplicação passa a ter autenticação própria, independente do IAM do Cloud Run, com **login por usuário e senha** (não OAuth) — decisão explícita do usuário, priorizando não depender de contas Google de terceiros e manter o cadastro de acesso sob controle total da aplicação.

**Abordagem técnica:** [Auth.js](https://authjs.dev) (NextAuth) com `Credentials Provider`, backed por uma tabela `users` própria no banco `arquitetura_planner` (senha com hash `bcrypt`, nunca em texto plano). Avaliada contra duas alternativas:

| Opção | Prós | Contras |
|---|---|---|
| Autenticação própria do zero (cookies/JWT + bcrypt manual) | Controle total, zero dependência nova | Time assume todo o risco de segurança de sessão/CSRF/timing attack — fácil errar detalhe sutil |
| **Auth.js + tabela `users` própria (escolhida)** | Biblioteca madura cuida de sessão/cookie/CSRF; dados do usuário continuam 100% na base própria; plugável — permite adicionar OAuth (Google ou outro) depois sem reescrever a camada de sessão | Exige aprender as convenções do Auth.js |
| Serviço terceiro (Auth0/Clerk/Supabase Auth) | Tela de admin pronta, menos código | Mais um fornecedor pago; dados de auth saem do Postgres próprio; contradiz a preferência por controle direto |

A escolha por Auth.js + tabela própria é o que garante a "evolução futura" pedida: adicionar um provedor OAuth mais tarde é configuração adicional no Auth.js, não uma reescrita da camada de sessão.

Cada ação relevante (gerar, editar, aprovar, mover atividade no board) passa a registrar o usuário autenticado que a realizou, substituindo o valor genérico `'user'` em `created_by`.

### 1.1 Tela de administração de usuários

Precisa de uma tela restrita a usuários com role `admin` para cadastrar novos usuários (nome, e-mail, senha inicial, role). Modelo de dados mínimo:

```sql
users(
  id, name, email UNIQUE, password_hash,
  role ('admin' | 'member'),
  created_at, updated_at
)
```

A tela de administração é um CRUD simples (`GET/POST/PATCH /users` no `planner-api`, protegido por checagem de role `admin` na própria API — não só escondido na UI). Não haverá autocadastro: só um admin pode criar novos usuários nesta primeira versão.

### 2. Hospedagem pública do frontend

Com autenticação de aplicação funcionando, o Cloud Run passa a aceitar tráfego público (`--allow-unauthenticated`), e a proteção de acesso migra do IAM de infraestrutura para o login da aplicação. Isso elimina a dependência do túnel `gcloud run services proxy` para uso normal.

> **Nota de implementação (15/07/2026):** a organização GCP do usuário tem uma política (`iam.allowedPolicyMemberDomains`) que bloqueia conceder `roles/run.invoker` para `allUsers` via IAM binding, mesmo sendo o único usuário/dono da organização. A alternativa usada foi `gcloud run services update arquitetura-planner --no-invoker-iam-check`, que desliga a checagem de IAM do Cloud Run para o serviço sem depender de um binding de `allUsers` — o `planner-api` continua exigindo IAM normalmente. Foi necessário também definir a variável `AUTH_URL` (URL pública do serviço) no Cloud Run: sem ela, o Auth.js resolve o `callbackUrl` do login usando o endereço interno do container (`http://0.0.0.0:8080`) em vez do domínio público, porque o Cloud Run não repassa o `Host` original do jeito que `req.nextUrl` do Next.js espera.

### 3. Persistência de sessão no navegador

Formulários com edição em andamento (ex.: `/planejamentos/[id]/editar`) não devem perder dados ao atualizar a página. Implementação via `localStorage`/`sessionStorage` no cliente, com limpeza automática após salvar com sucesso.

### 4. Drag-and-drop no board

O board (ADR-004) troca os controles de botão/seletor por arrastar-e-soltar de verdade para mover atividades entre colunas.

### 5. Redesign visual

Novo design system aplicado a toda a aplicação (não só ao board), na linha "vidro líquido" (glassmorphism) inspirada no macOS recente: fundo com gradiente suave, cards e colunas com efeito de vidro translúcido (`backdrop-blur`), cantos arredondados generosos, microinterações de hover (leve escala e brilho ao passar o mouse), botões com aparência de "pílula de vidro".

## Riscos e considerações

- **Sem autocadastro:** só um admin pode criar usuários (tela de administração, item 1.1). Reduz superfície de ataque ao ficar público, mas significa que perder acesso ao único admin trava a criação de novos usuários — mitigação: garantir pelo menos um admin criado via seed/migration na primeira implantação, não só pela UI.
- **Custo e abuso:** expor publicamente um serviço Cloud Run tem risco de tráfego indesejado se a URL vazar. Mitigação: login continua obrigatório mesmo com o serviço público; considerar rate limiting básico no endpoint de login.
- **Volume de trabalho do redesign:** é o item de maior esforço de front-end desta ADR e deve ser tratado como sub-fase própria da implementação, não misturado com autenticação/sessão no mesmo commit.
- **Migração dos dados existentes:** planejamentos e versões já criados têm `created_by = 'user'` genérico — não há como atribuir retroativamente ao usuário real; aceito como limitação conhecida.

## Fora de escopo

Multi-tenancy completo (organizações isoladas entre si); fluxo de recuperação de senha por e-mail (na primeira versão, reset de senha é feito por um admin diretamente na tela de administração); autocadastro de novos usuários; customização de tema por usuário; qualquer mudança em `planner-api` além de registrar o usuário autenticado nas gravações já existentes e do CRUD de `users`.

## Critérios de sucesso

- Login por usuário e senha protege o acesso à aplicação sem depender de túnel `gcloud`.
- Um admin consegue cadastrar, editar e desativar usuários pela tela de administração.
- Cada ação relevante registra o usuário autenticado que a realizou.
- Atualizar o navegador durante uma edição não perde os dados preenchidos.
- Atividades podem ser movidas entre colunas do board por arrastar-e-soltar.
- O novo design system é aplicado de forma consistente nas telas principais (formulário de geração, histórico, detalhe, edição, board).
